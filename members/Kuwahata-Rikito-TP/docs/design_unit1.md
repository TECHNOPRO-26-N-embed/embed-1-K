## 1. システム全体像

### 1-1. ブロック構成（入力 -> 処理 -> 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                     【Arduino UNO R3】              【出力】
超音波センサー(HC-SR04) ─→┐                             ┌──→ パッシブブザー
                         ├ 距離計測・3段階判定・状態管理 ├──→ 4桁7セグ表示
　　　　　　　　　　　　　　                             └──→ （任意）LED
```

補足説明（図で表現しきれない場合）：
- 距離は100ms周期で更新し、近・中・遠の状態にマッピングする。
- 音出力と表示更新は同一ループ内で並行処理するため、ブロッキング処理を避ける。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（ピン初期化と表示初期化完了）
[待機中] ──（有効距離を取得）──→ [距離判定中] ──（区分確定）──→ [通知中]
   ↑                                 │                              │
   └────────────（異常値継続終了/監視継続）──────────────┴──（次周期100ms）

[距離判定中] ──（異常値検出）──→ [異常監視]
[異常監視] ──（有効値復帰）──→ [距離判定中]
```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 待機中 | センサー計測タイミング待ち、前回値保持 | ブザー停止、表示は前回値維持 |
| 距離判定中 | 取得した距離を近/中/遠に分類 | 出力はまだ更新しない |
| 通知中 | 区分に応じて音パターンと表示を更新 | 近:高音連続 / 中:中音断続 / 遠:低音短音 |
| 異常監視 | 範囲外値（0cmや400cm超）を監視 | ブザー停止、表示は前回正常値維持 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| 距離読み取り | 100ms | **millis** | 常時監視しつつ他処理も動かすため |
| 中距離の断続音 | 100ms ON / 100ms OFF | **millis** | 表示更新と並行するため |
| 表示更新 | 100ms | **millis** | 応答時間0.2秒以内を満たすため |
| 起動時確認音 | 100ms 1回 | **delay** | setup()内のみで使用し通常処理に影響しないため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

### 3-1. 部品一覧とピン接続

> ※ デジタルピン: D2〜D13 / アナログピン: A0〜A5

| 部品名 | 型番・種類 | Arduinoピン | 入力 / 出力 | 備考 |
|:--|:--|:--|:--|:--|

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| センサー異常値 | 0mmや4000mm超が返る | 計測値が仕様範囲外 | 値を無視して前回有効値を使用、連続時はブザー停止 |
| 計測タイムアウト | Echoが返らない | pulseInのタイムアウト判定 | 異常監視状態へ遷移し次周期で再計測 |
| 表示器未接続 | TM1637通信失敗 | 表示更新時に不正値が継続 | シリアルへエラー表示し音通知は継続 |
| ノイズによる境界揺れ | しきい値付近で値が揺れる | 連続計測で区分が頻繁に反転 | ヒステリシス（±20mm）を適用 |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- 現時点でピン競合はない。
- TM1637ライブラリとtone()の併用は可能だが、更新周期が短すぎると表示ちらつきや音途切れが出る可能性がある。
- 状態遷移は異常監視から復帰ルートがあり妥当。
- loop()内のdelay多用は避けるべき。

**対応した内容：**
- タイミング設計をmillis()中心に統一した。
- setup()以外ではdelayを使わない方針を明記した。

---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- 本設計はボタン主体ではないためチャタリング影響は小さい。
- delay常用は応答遅延の原因になる。
- 超音波でも境界付近の揺れ対策（ヒステリシス等）が有効。

**対応した内容：**
- 異常系にしきい値境界ノイズ対策（ヒステリシス）を追加した。
- delay使用箇所を起動時確認音のみとした。
