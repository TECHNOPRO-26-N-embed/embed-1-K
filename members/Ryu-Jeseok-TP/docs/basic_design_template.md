## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】            【出力】
IRリモコン受信 ─────→┐                         ┌──→ パッシブブザー
                     ├── 時刻管理・状態遷移 ───┤
                     ├── 問題判定・入力検証 ───┼──→ RGB LED
                     └── ノンブロッキング制御 ─┴──→ 16x2 LCD(I2C)




```
補足説明（図で表現しきれない場合）：

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[時刻設定待ち] ──（現在時刻入力完了）──→ [待機中] ──（アラーム時刻到達）──→ [出題中]
   ↑                     │                             │（正解）
   └──（設定取消/受信失敗復帰）[入力エラー] ←─（不正入力/受信失敗）┘
                              └──────────────→ [待機中]


```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 時刻設定待ち | 現在時刻(HHMM)とアラーム時刻(HHMM)を順に受け付ける | LED青固定、ブザー停止、LCDに入力ガイド表示 |
| 待機中 | `millis()`で時刻更新しアラーム到達判定を行う | LED緑ゆっくり点滅、ブザー停止 |
| 出題中 | 問題文表示、リモコン回答入力、正誤判定を繰り返す | LED赤点灯、ブザー周期鳴動、LCDに問題と入力値表示 |
| 入力エラー | 範囲外時刻・不正キー・受信失敗を通知し再入力へ戻す | LED黄点滅、エラー種別別の短い警告音 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| IR入力監視 | 20ms程度 | **millis** | 入力遅延を抑え2秒以内の応答を満たすため |
| アラーム鳴動ON/OFF切替 | 200ms | **millis** | 鳴動中でも回答入力を止めないため |
| 現在時刻進行 | 60000ms | **millis** | 1分刻み更新をノンブロッキングで行うため |
| LCD再描画 | 100〜200ms | **millis** | 表示更新と他処理を並行するため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| リモコン連打/長押し誤判定 | 同じキーを短時間で連続受信したとき | 前回キー受信から50ms未満の再受信を検出 | デバウンス処理で重複入力を無視 |
| 時刻入力範囲外 | HH>23 または MM>59 を入力したとき | 確定時に範囲チェック | errorType=1、警告音後に同項目を再入力 |
| IR受信失敗 | 一定時間キーを受信できない/デコード失敗 | タイムアウト・decode結果判定 | errorType=2、入力待ち継続とLCDに再操作案内 |
| 不正キー入力 | 数字入力中に許可外キーを押したとき | キーコードを許可表と照合 | errorType=3、短音を鳴らし入力は破棄 |
| 問題の誤答 | 出題中に不正解を入力 | 正解データとの比較不一致 | アラーム継続、次回入力を促す |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合は見当たらない（D0/D1未使用、LCDはA4/A5固定）。
- 主要リスクは IRremote と tone() のタイマー干渉で、鳴動中の受信低下に注意が必要。
- 状態遷移は「入力エラー→時刻設定待ちへ復帰」を明示したため、行き止まりはない。
- `delay()`を使うと回答入力が止まるため、全処理を`millis()`で更新する方針が妥当。

**対応した内容：**
- ブザーは短周期ON/OFFを`millis()`で制御する設計に統一。
- エラー復帰ルートを状態遷移図へ追記。
- ライブラリ競合対策として、鳴動パターンを単純化（同時多重音を使わない）。

---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- 不正入力対策（時刻範囲チェック、許可キー判定）は設計に含まれている。
- `delay()`不使用のため、鳴動中でも入力受付を継続できる。
