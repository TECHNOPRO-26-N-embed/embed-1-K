### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON/初期化]
  ↓（初期化完了）
[待機] ──（ICカード検出）──→ [真偽判定] ──（判定結果）──→ [結果出力]
   ↑                                            │
   │                                            │
   │←─────（エラー発生）──────[エラー]←────┘
   │                                            │
   │←────（タイムアウト）────[タイムアウト]←┘
   │                                            │
   └───────────────（出力/エラー/タイムアウト終了）──────────────┘
```



**各状態の意味（復帰経路も明記）：**

| 状態名      | この状態のとき何をしているか                | 出力の様子                | 復帰経路（次に遷移する状態）        |
|:------------|:------------------------------------------|:--------------------------|:--------------------------------|
| 待機        | ICカードやリモコン入力を監視し続ける（1.5s毎）| なし                      | 入力検出で「真偽判定」へ            |
| 真偽判定    | 入力データの判定処理を行う                 | なし                      | 判定結果で「結果出力」または「エラー」|
| 結果出力    | 判定結果に応じてLCD表示・ブザー鳴動         | LCD表示・ブザー音           | 一定時間後または操作で「待機」へ    |
| エラー      | 異常発生時のエラー表示・警告音              | LCDにエラー表示・警告音     | 一定時間後または操作で「待機」へ    |
| タイムアウト| 入力や処理が規定時間内に完了しなかった場合  | LCDにタイムアウト表示・警告音| 一定時間後または操作で「待機」へ     |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| ICカード読み取り | 200ms | **millis** | 常時監視が必要なため |
| LCD表示更新 | 500ms | **millis** | ちらつき防止と負荷軽減のため |
| ブザー音出力 | 100ms | **millis** | IR受信とのタイマー干渉を避け、非ブロッキングで動作させるため |
| 音分岐制御 | 100ms | **millis** | toneを使わずdigitalWrite制御で入力取りこぼしを防ぐため |
| 状態遷移監視 | 200ms | **millis** | 状態変化をスムーズに検知するため |
| エラー再試行 | 1秒 | **millis** | 再試行間隔を確保し、負荷を軽減するため |
| 初期化完了待機 | 1秒 | **delay** | 起動時の安定性を確保するため |
| 赤外線リモコン受信 | 100ms | **millis** | コマンドを確実に受信するため |
| パスワード入力タイムアウト | 10秒 | **millis** | ユーザー操作の待機時間を設定するため |
| リモコン長押し判定 | 500ms | **millis** | 長押しと単押しを区別するため |
| リモコン再送信間隔 | 200ms | **millis** | 誤動作防止と負荷軽減のため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| RFID未検出 | 読み取り時間内にカードがかざされない | 1.5秒タイムアウト | LCDに NO CARD を表示し、短い警告音を鳴らして待機状態へ戻る |
| RFID読み取り失敗 | カードはあるがデータ取得に失敗する | ライブラリ戻り値がエラー | 最大3回リトライし、失敗時は READ ERR を表示して処理を中断する |
| RFID書き込み失敗 | 書き込み処理が完了しない | 書き込みAPI戻り値が失敗 | 1回だけ再試行し、失敗時は WRITE ERR を表示して失敗音を鳴らす |
| 赤外線コマンド不正 | 定義外のリモコンコードを受信する | コマンドテーブル照合で未定義 | その入力を無視し、画面と音の状態を変更しない |
| パスワード不一致 | 入力値が登録パスワードと異なる | 文字列比較で不一致 | 残り試行回数を表示し、規定回数超過で30秒ロックする |
| ブザー停止漏れ | 異常時にブザーが鳴り続ける | 鳴動時間が上限を超過 | 最大鳴動時間で強制停止し、エラーコードを記録する |
| 状態遷移ハング | 処理状態から復帰しない | 状態滞在時間が上限を超過 | タイムアウトで待機状態へ強制復帰し、SYSTEM ERR を表示する |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：** 
ピン競合は現状なし（D0/D1未使用、SPIとI2Cの使い分けも適切）。ただしRFIDのRSTピンは明示が必要。
ライブラリは基本共存可能だが、IR受信とブザー制御でタイマー干渉の可能性あり。
状態遷移は「結果出力/エラー/タイムアウトから待機へ戻る」経路を明示すると安全。
`delay()`は入力取りこぼしの原因になるため、運用時は`millis()`中心が望ましい。

**対応した内容：** 
RSTピンをD9へ設定した。
ブザー制御は第一候補としてmillisとdigitalWriteの非ブロッキング方式へ統一し、toneは使用しない方針に修正した。
経路をめいじした　
---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
この設計では物理ボタン入力を使っていないため、チャタリング対策は現状ほぼ不要（ボタン追加時はデバウンスが必要）。
`delay()`はブザー音出力や初期化待機で使うと、その間にRFID監視・赤外線受信・状態更新が止まり、取りこぼしや応答遅延の原因になる。
アナログセンサーは現時点で未使用のためノイズ対策は必須ではないが、将来追加時は移動平均・閾値ヒステリシス・異常値除外が必要。

**対応した内容：**
運用中のdelayを見直し、ブザー音出力と音分岐制御はmillisベースへ変更した。IR受信との干渉回避のためtoneは使わずdigitalWriteで制御する方針にした。
