## 1. グローバル変数・定数の設計

```
【ピン定義】
PIN_POT        : const uint8_t = A1
PIN_MOTOR_PWM  : const uint8_t = 3
PIN_MOTOR_D1   : const uint8_t = 4
PIN_MOTOR_D2   : const uint8_t = 5
PIN_LED_1      : const uint8_t = 6
PIN_LED_2      : const uint8_t = 7
PIN_LED_3      : const uint8_t = 8
PIN_LED_4      : const uint8_t = 9
PIN_LED_5      : const uint8_t = 10
PIN_BUZZER     : const uint8_t = 12

LED_PINS       : const uint8_t[5] = {6, 7, 8, 9, 10}

【スケール定数】
WIND_MIN       : const int = 0
WIND_MAX       : const int = 2500

【入力・制御値】
potValue       : uint16_t = 0          // 0〜1023
windValue      : int = 0               // 0〜2500
motorPwm       : uint8_t = 0           // 0〜255

【ブザー制御用】
buzzerPatternActive : bool = false
buzzerPhase         : uint8_t = 0
buzzerPhaseStartMs  : unsigned long = 0
```

> 備考: 実装は状態変数（待機/計測/警告）を持たず、`windValue` から都度 `level` を算出して出力を更新する方式。

## 2. 各関数の詳細設計

### `setup()` — 初期化処理

```
【処理の流れ】
1. PIN_POT を INPUT、モーター/LED/ブザーの各ピンを OUTPUT に設定する。
2. モーター方向を右回転固定に設定する（D1=HIGH, D2=LOW）。
3. PWM=0、LED全消灯、ブザーOFFで安全な初期状態を作る。
```

### `loop()` — メインループ

```
【処理の流れ】
1. ポテンショメータを読み取り、0〜1023に制限して potValue に格納する。
2. potValue を 0〜2500 にマッピングして windValue を算出する。
3. windValue を 0〜255 にマッピングして motorPwm を算出する。
4. モーター方向を右回転固定で再設定し、PWMを反映する。
5. showWindLevelOnLeds(windValue) でLED段階表示を更新する。
6. getWindLevel(windValue) の結果を updateBuzzer(level) に渡して警告音を制御する。
7. delay(20) でループ周期を安定化する。
```

### `turnOffAllLeds()` — LED全消灯

```
【処理の流れ】
1. LED_PINS 配列を先頭から走査する。
2. 5本すべてのLEDに LOW を出力する。
```

### `getWindLevel(int value)` — 風量値を5段階へ変換

```
【処理の流れ】
1. value を WIND_MIN〜WIND_MAX で constrain する。
2. clamped==0 のとき level=0 を返す。
3. 0超〜500:1、〜1000:2、〜1500:3、〜2000:4、〜2500:5 を返す。
```

### `showWindLevelOnLeds(int value)` — LED段階表示

```
【処理の流れ】
1. getWindLevel(value) で level を得る。
2. turnOffAllLeds() で全消灯する。
3. level が 0 なら終了する。
4. LED1 から LED(level) まで HIGH を出力して段階点灯する。
```

### `updateBuzzer(uint8_t level)` — 断続ブザー制御

```
【処理の流れ】
1. level != 5 のとき:
	- ブザーをOFF、パターン状態を初期化して終了する。
2. level == 5 のとき:
	- 初回のみパターン開始（phase=0, 開始時刻記録, ブザーON）。
3. 2回目以降は millis() で経過時間を確認し、位相を進める。
	- phase 0: ON 120ms
	- phase 1: OFF 120ms
	- phase 2: ON 120ms
	- phase 3: OFF 600ms
4. 位相更新時にON/OFFを切り替える。
```

## 3. タイミング設計の具体化

### 3-1. `delay(20)` を使う処理

```
【対象】
- ポテンショメータ読取
- 風量/ PWM再計算
- モーター出力更新
- LED段階表示更新

【設計意図】
- 1ループ約20msで更新して操作感を確保する。
- 過度な高速更新を避け、値の暴れを抑える。
```

### 3-2. `millis()` を使う処理

```
【対象】
- 最大風量時のブザー断続パターン

【処理の流れ】
1. now = millis() を取得。
2. 現在phaseに対応する継続時間(120/120/120/600ms)を決める。
3. now - buzzerPhaseStartMs が継続時間以上なら次phaseへ進める。
4. phaseに応じてブザー出力をON/OFFする。
```

### 3-3. その他の重要ロジック

```
【風量と表示/警告の対応】
- windValue: 0          -> level 0（全LED消灯、ブザーOFF）
- windValue: 1〜500     -> level 1
- windValue: 501〜1000  -> level 2
- windValue: 1001〜1500 -> level 3
- windValue: 1501〜2000 -> level 4
- windValue: 2001〜2500 -> level 5（ブザー断続ON）
```

## 6. AIレビュー記録

### Q1: 実装上の問題確認

> 「この詳細設計書に書いた関数と処理フローをもとに Arduino でコードを書きます。バグになりやすい箇所・処理の抜け・型の問題はありますか？」

**AIの回答（要約）：**
- 高: `getWindLevel()` の閾値境界（500/1000/1500/2000）でオフバイワンが起きやすい。
- 中: `updateBuzzer()` の位相管理で、`level != 5` 復帰時に状態初期化を忘れると鳴りっぱなしになる。
- 中: `delay(20)` により将来の機能追加時に応答遅延が顕在化する可能性がある。

**対応した内容：**
- 境界値テストを Section 5-1 に追加した。
- ブザー解除時の初期化仕様を Section 2 に明記した。
- `delay(20)` 採用理由と制約を Section 3 に明記した。

### Q1: 実装上の問題確認

> 「この詳細設計書に書いた関数と処理フローをもとに Arduino でコードを書きます。バグになりやすい箇所・処理の抜け・型の問題はありますか？」

**AIの回答（要約）：**
1.高: 状態定義が基本設計と不一致です（型と状態名の両方）。
  状態定義 が int / 0:待機 1:動作中 2:警告中 になっていますが、基本設計は byte / 0:待機 1:計測 2:警告です。
  参照: 基本設計の状態
  影響: 状態遷移ロジック、テスト期待値、デバッグログの解釈がズレます。

2.高: テンプレ残骸が実仕様と混在しています。
  setup例 に PIN_BUTTON / PIN_LED が残り、
  loop例 に完了/エラー状態が残っています。
  影響: 実装者が誤って旧テンプレに従うリスクが高いです。

3.中: 3-1が「設定を行わない」と書かれているのに、内容はボタンのデバウンス例のままです。
  参照: 3-1設定なし宣言, 3-1内容
  影響: 「何を採用しないのか」「代替ロジックは何か」が不明です。

4.中: SRAM合計が 1B になっており、現行の変数量と整合していません。
  参照: SRAM記載
  影響: リソース見積もりの信頼性が下がります。

5.中: 表や文に誤記・ノイズがあり、テスト仕様の可読性を下げています。
  例: No重複, 文言崩れ, ザー停止
  影響: レビュー時の誤解や記入ミスを誘発します。

**対応した内容：**
グローバル変数・定数の型と初期値が明記され、SRAM合計も現実的な値（16B）に修正済み
状態管理（currentState）はbyte型で0:待機/1:動作中/2:警告中と明確化
テンプレ残骸（PIN_BUTTON, PIN_LED, 完了/エラー状態）は削除、実装対象の状態・ピンのみ記載
